     1→# Fitn — Flutter App Specification
     2→
     3→**Product:** Fitn — a mobile fitness companion.
     4→**Platform:** Flutter (iOS + Android), single Dart codebase.
     5→**Audience:** Senior Flutter engineer. Build-ready.
     6→
     7→This document specifies a fitness app built around a deterministic, physiology-grounded engine. The engine is the product's core value — its formulas, thresholds, and decision trees are pinned down completely. Everything around it (UI, state, storage, sync) is specified to a buildable level of detail, with implementation choices made where they matter and left to the engineer where they don't.
     8→
     9→---
    10→
    11→## 1. Product Vision
    12→
    13→A mobile fitness companion that turns a 5-minute onboarding questionnaire into a fully personalized, deterministic fitness plan — body composition assessment, nutrition targets, multi-week training program, 7-day meal rotation — and then **stays with the user daily**: logging workouts, tracking progress, and tuning the plan as their body changes.
    14→
    15→**The differentiator is the engine.** It's deterministic (same inputs → byte-identical output), grounded in published physiology (Mifflin-St Jeor, Katch-McArdle, Kouri FFMI, Hodgdon-Beckett Navy, CUN-BAE, ABSI, RippedBody volume landmarks, Lyle McDonald muscle-gain rates), and produces a coherent cut/bulk/recomp/maintenance/reverse-diet recommendation with a defensible rationale. The app's job is to get out of the engine's way.
    16→
    17→**What the app must do well:**
    18→1. **Onboard in 5 steps** — minimal friction, sensible defaults, never block on optional fields.
    19→2. **Generate a plan in <500 ms** — instant feedback, no spinner anxiety.
    20→3. **Work offline** — the plan, exercise library, and recipe database live on-device. Sync is a background concern, never a UX blocker.
    21→4. **Log workouts** — sets, reps, weight, RPE. Rest timer. This is the daily-use loop.
    22→5. **Track progress** — weight log, workout history, volume trends. The engine accepts weight/intake logs as inputs for adaptive TDEE; the UI must collect them.
    23→6. **Sync across devices** — sign in, pick up where you left up on another phone.
    24→7. **Regenerate gracefully** — when the user's body changes, the plan should update without losing history.
    25→
    26→**What the app must NOT do:**
    27→- Block on auth. Anonymous use is first-class.
    28→- Require a server round-trip for plan generation.
    29→- Lose data on sign-out (local data survives; only server-synced data is disowned).
    30→- Pretend to be a marketplace. Commerce is out of scope.

    34→## 2. Key Decisions
    35→
    36→### 2.1 In scope
    37→
    38→| Feature                               | Notes                                                                                            |
    39→| ------------------------------------- | ------------------------------------------------------------------------------------------------ |
    40→| On-device engine in a Dart isolate    | Plan generation is instant, offline, free. Runs via `Isolate.run()`; engine lives in a pure-Dart `fitn_engine` package. |
    41→| 5-step onboarding                     | Minimal friction, sensible defaults, optional fields never block.                               |
    42→| 5 main tabs                           | Home, Workouts, Meals, Progress, Profile.                                                       |
    43→| Workout session logger                | Full-screen set/rep/weight/RPE logger with rest timer. Every set persists to `WorkoutLogRecord`. |
    44→| Weight + intake logging               | Engine inputs for adaptive TDEE — collecting them closes the feedback loop. Stored as `WeightLogRecord` / `IntakeLogRecord`. |
    45→| Progress tracking                     | Weight chart, volume trends, workout history, PRs (computed from workout logs).                 |
    46→| Exercise library browser              | Browse all 1,217 exercises independently of a plan (search, filter by muscle/equipment).        |
    47→| Plan versioning + restore             | History of generated plans with rollback.                                                       |
    48→| Background sync queue                 | Optimistic local writes, `workmanager`-driven flush, `updated_at` conflict resolution.          |
    49→| Anonymous-first auth                  | Plan works without sign-in. Optional Supabase auth (magic-link + Google/Apple OAuth) for cross-device sync. |
    50→| Single program plan type              | Every plan is a program (≥1 mesocycle). No user-facing "standard vs program" toggle.            |
    51→| Diet types: omnivore, vegetarian, vegan | Three supported diets. Engine rejects keto/paleo; UI does not expose them.                     |
    52→| YouTube exercise videos               | Embedded via `youtube_player_flutter`. No local video hosting.                                  |
    53→
    54→### 2.2 Out of scope (deferred)
    55→
    56→| Item                                  | Why                                                                                              |
    57→| ------------------------------------- | ------------------------------------------------------------------------------------------------ |
    58→| Custom auth tables (`Account`, `Session`, `VerificationToken`) | Supabase Auth manages these internally.                                                         |
    59→| Marketplace / commerce (`Product`, `WishlistItem` tables, checkout) | No payment integration. Build fresh when prioritized.                                          |
    60→| Keto / paleo diet support             | Engine doesn't model them. Adding requires nutrition + meal plan rework.                        |
    61→| Local video file hosting              | YouTube embeds cover the use case; bundling MP4s bloats the app.                                 |
    62→| Realtime cross-device sync            | Pull-on-foreground + push-on-write is sufficient for launch. Supabase Realtime available later. |
    63→| Edge Functions / server-side logic    | Not needed — all logic runs on-device.                                                           |
    64→| Social features (sharing, leaderboards) | Not part of the single-user fitness companion model.                                           |
    65→| Custom exercise / recipe creation     | Engine uses fixed libraries (1,217 exercises, ~400 recipes). Schema changes required to support. |
    66→| Wear OS / watchOS companion           | Future enhancement for rest timer + set logging from the wrist.                                 |
    67→| Apple Health / Google Fit sync        | Future enhancement for pulling weight + workouts from health APIs.                              |
    68→| Adaptive plan adjustments UI          | Engine has `recommendCutAdjustment` / `recommendBulkAdjustment` (plateau detection + macro tuning). Wire into UI when users have 3+ weeks of weight logs. |

    91→
    92→### 2.4 Why Supabase
    93→
    94→Supabase provides Postgres, Auth (magic-link + OAuth), Realtime, and Storage as one service with one Flutter SDK. For a mobile app this avoids deploying, monitoring, and scaling a custom backend. The data model is plain Postgres with row-level security (RLS), so the data is portable — Supabase is a hosting layer, not a proprietary lock-in.
    95→
    96→Supabase provides:
    97→- **Postgres** with row-level security — no server code needed for CRUD.
    98→- **Auth** with magic-link, Google, Apple, GitHub out of the box.
    99→- **Realtime** (optional — cross-device live sync; not needed for launch).
   100→- **Storage** for any future self-hosted media (not needed — YouTube embeds suffice).
   101→- **Edge Functions** if server-side logic is ever needed (not needed for launch).

   135→### 3.2 Engine orchestration
   136→
   137→```mermaid
   138→flowchart LR
   139→  Input["UserProfile + PlanPreferences"] --> Assess["assessProfile()"]
   140→  Assess --> AR["AssessmentResult<br/>(is_partial flag)"]
   141→  AR -->|"is_partial == false"| Plan["proposePlan()"]
   142→  Input --> Plan
   143→  Plan --> NP["buildNutritionPlan()"]
   144→  Plan --> TP["buildTrainingPlan()"]
   145→  Plan --> MP["buildMealPlan()"]
   146→  NP --> FP["FitnessPlan"]
   147→  TP --> FP
   148→  MP --> FP
   149→  AR -->|"is_partial == true"| Err["PartialAssessmentError<br/>(UI surfaces regenerate CTA)"]
   150→```
   151→
   152→**Critical contract:** `proposePlan()` refuses to run on a partial assessment. The UI must check `assessment.isPartial` first and show a "regenerate" CTA instead of attempting plan generation.
   153→
   154→### 3.3 Plan generation lifecycle
   155→
   156→```mermaid
   157→sequenceDiagram
   158→  participant U as User
   159→  participant UI as Flutter UI
   160→  participant S as Riverpod State
   161→  participant E as fitn_engine (Isolate)
   162→  participant IDB as Isar
   163→  participant Q as Sync Queue
   164→  participant SB as Supabase
   165→
   166→  U->>UI: Completes onboarding step 5
   167→  UI->>S: generatePlan(profile, prefs)
   168→  S->>S: setGenerating(true)
   169→  S->>E: Isolate.run(() => proposePlan(...))
   170→  E-->>S: FitnessPlan (50-200 ms)
   171→  S->>IDB: PUT plan (active=true), PUT profile
   172→  S->>S: setPlan(plan), setOnboarded(true)
   173→  S-->>UI: notify → render Home tab
   174→  S->>Q: enqueue({type: plan, data})
   175→  Q->>SB: UPSERT plan (when online)
   176→  SB-->>Q: 200 OK
   177→  Q->>IDB: mark synced
   178→```
   179→
   180→Generation is local and synchronous from the UI's perspective (the isolate is an implementation detail). Sync is queued and flushed in the background. Failures are retried with exponential backoff, never shown to the user unless they persist for >24h.
   181→
   182→### 3.4 Sync protocol
   183→
   184→- **Optimistic local writes** — every mutation writes to Isar immediately with `syncStatus: pending`.
   185→- **Background flush** — `workmanager` runs every 15 min (when online), drains the queue.
   186→- **Conflict resolution** — `updated_at` timestamp. Server wins if `server.updated_at > local.updated_at`. Last-writer-wins is fine for this domain (single user, single device active at a time in practice).
   187→- **Push** — `UPSERT` to Supabase with `WHERE user_id = auth.uid()` (RLS enforces ownership).
   188→- **Pull** — on sign-in, fetch all rows for the user, merge by `updated_at`.
   189→- **No realtime at launch** — cross-device live sync is nice-to-have, not required. Pull-on-foreground + push-on-write is sufficient.

   193→## 4. Engine (`fitn_engine` package)
   194→
   195→Pure Dart. No Flutter. No I/O. JSON data files loaded from `assets/` at construction.
   196→
   197→**Public API:**
   198→
   199→```dart
   200→// fitn_engine/lib/fitn_engine.dart
   201→
   202→class FitnEngine {
   203→  FitnEngine({required EngineData data});  // data loaded by caller
   204→
   205→  AssessmentResult assessProfile(UserProfile profile);
   206→  FitnessPlan proposePlan(UserProfile profile, AssessmentResult assessment, PlanPreferences prefs);
   207→  GeneratePlanResponse generatePlan(UserProfile profile, PlanPreferences prefs);  // convenience: assess + propose
   208→}
   209→
   210→class EngineData {
   211→  final List<Exercise> exercises;        // 1,217
   212→  final List<SplitDesign> splits;        // 8
   213→  final Map<String, MovementPatternSpec> movementPatterns;  // 40
   214→  final List<Recipe> recipes;            // ~400 (curated + uncurated + pre/post)
   215→  final Map<String, FoodItem> foodDatabase;
   216→}
   217→
   218→// Loads from rootBundle (Flutter) or File (CLI/test)
   219→Future<EngineData> loadEngineData({String? basePath});
   220→```

   337→#### 4.1.3 Outputs
   338→
   339→The full output tree (`AssessmentResult`, `BodyComposition`, `HealthRiskAssessment`, `MuscularPotential`, `NutritionPlan`, `RMRResult`, `TDEEResult`, `CalorieTargets`, `MacroSplit`, `HydrationTarget`, `MicronutrientTargets`, `TrainingPlan`, `Mesocycle`, `Microcycle`, `Workout`, `WorkoutExercise`, `Exercise`, `MealPlan`, `DayPlan`, `Meal`, `Recipe`, `MealFood`, `FoodItem`, `FitnessPlan`) is defined in §9.1 (Output model reference).
   340→
   341→### 4.2 Assessment sub-engine
   342→
   343→Orchestration (`assessProfile`) runs 4 sub-assessments in order, **never short-circuiting on failure**:
   344→
   345→1. `assessBodyComposition(profile)` — root of the dependency tree.
   346→2. `assessHealthRisk(profile)` — independent of body comp.
   347→3. `assessMuscularPotential(profile, bodyComp.bodyFatPct)` — **skipped** if `bodyComp == null`.
   348→4. `decideStrategy(profile, bodyComp.bodyFatPct)` — **skipped** if `bodyComp == null`, defaults to `maintenance`.
   349→
   350→If any sub-assessment throws, record `"sub_name: ArgumentError: <msg>"` in `errors` and set the sub-result to null. `is_partial = errors.isNotEmpty`. The summary string is built defensively (every nullable field guarded).
   351→
   352→`proposePlan()` throws `PartialAssessmentError` if called with `is_partial == true`.
   353→
   354→#### 4.2.1 Body fat % selection (priority order)
   355→
   356→1. **User-provided** (`bodyFatPct != null`) → use as-is.
   357→2. **US Navy circumference** (Hodgdon & Beckett 1984) — if `hasCircumferenceMeasurements`:
   358→   - Male: `86.010 × log10(waistIn − neckIn) − 70.041 × log10(heightIn) + 36.76`
   359→   - Female: `163.205 × log10(waistIn + hipIn − neckIn) − 97.684 × log10(heightIn) − 78.387`
   360→   - If `(waistIn − neckIn) ≤ 0` (male) or `(waistIn + hipIn − neckIn) ≤ 0` (female), return null → fall through to CUN-BAE.
   361→   - Clamp to `[2.0, 60.0]`.
   362→3. **CUN-BAE** (always computable):
   363→   - `bf = -24.988 + 0.503 × age + 1.0689 × bmi + 0.462 × (sex == female ? 1.0 : 0.0)`
   364→   - Clamp to `[2.0, 60.0]`.
   365→

   367→
   368→| Category     | BMI range            |
   369→| ------------ | -------------------- |
   370→| UNDERWEIGHT  | `[0, 18.5)`          |

   378→| ------------ | -------------- | --------------- |
   379→| ESSENTIAL    | `[0, 6)`       | `[0, 14)`       |

   384→
   385→#### 4.2.4 FFMI (Kouri 1995)
   386→
   387→- `ffmKg = weightKg × (1 − bfPct/100)`
   388→- `ffmi = ffmKg / heightM²`
   389→- `normalizedFfmi = ffmi + 6.1 × (1.8 − heightM)`
   390→
   391→**Ceiling constants:** `FFMI_NATURAL_COMMON = 25.0`, `ATTAINABLE = 27.3`, `LIKELY_MAX = 28.0`. Normalization coefficient `6.1`, reference height `1.8 m`.
   392→
   393→**Named target BF% landmarks** (for `targetWeightsKg` map):
   394→
   395→| Label            | Male | Female |
   396→| ---------------- | ---- | ------ |
   397→| athletic         | 14.0 | 21.0   |
   398→| fitness          | 18.0 | 25.0   |
   399→| acceptable       | 25.0 | 32.0   |
   400→| hormonal_floor   | 10.0 | 18.0   |
   401→
   402→`targetWeight = lbm / (1 − targetBf/100)` where `lbm = currentWeight × (1 − currentBf/100)`.
   403→
   404→#### 4.2.5 Health risk
   405→
   406→| Metric | Formula | Classification |
   407→| ------ | ------- | -------------- |
   408→| **WHR** | `waist / hip` | Male: ≤0.85 LOW, ≤0.90 MOD, ≤1.0 HIGH, >1.0 VERY_HIGH. Female: ≤0.80 LOW, ≤0.85 MOD, ≤1.0 HIGH, >1.0 VERY_HIGH. |
   409→| **WHtR** | `waist / height` | Male: <0.50 LOW, <0.53 MOD, <0.58 HIGH, ≥0.58 VERY_HIGH. Female: <0.46 LOW, <0.49 MOD, <0.54 HIGH, ≥0.54 VERY_HIGH. |
   410→| **ABSI** | `(waist_m) × weight^(-2/3) × height^(5/6)` | Z-score from NHANES 10-year age bands (see §9.3). |
   411→| **IBW** | `base + mult × (heightIn − 60)` | 4 formulas: Devine (50/2.3 M, 45.5/2.3 F), Robinson (52/1.9, 49/1.7), Miller (56.2/1.41, 53.1/1.36), Hamwi (48/2.7, 45.4/2.2). |
   412→
   413→**Overall risk** (weighted heuristic): ABSI weight 0.5, WHR 0.3, WHtR 0.2. Add weight when the metric is HIGH/VERY_HIGH (ABSI: ABOVE_AVERAGE/HIGH). Map: ≥0.75 → VERY_HIGH, ≥0.50 → HIGH, any clinical risk factor → MODERATE, else LOW.
   414→
   415→#### 4.2.6 Muscular potential
   416→
   417→- Use **normalized FFMI** for ceiling comparison.
   418→- `ffmiToCeilingPct = clamp(normalizedFfmi / 25.0 × 100, 0, 100)`.
   419→- `headroomKg = max(0, ffmAtCeiling − ffmKg)` where `ffmAtCeiling = (25.0 − 6.1 × (1.8 − heightM)) × heightM²`.
   420→- **Berkhan stage max** (men only): `heightCm − 100` at 5-6% BF. Null for women (essential BF floor is ~10-13%, not 5-6%).
   421→- **Expected monthly muscle gain** (Lyle McDonald, men; × 0.5 for women):
   422→
   423→| Status        | Monthly gain (men) |
   424→| ------------- | ------------------ |
   425→| BEGINNER      | 0.85 kg            |
   426→| NOVICE        | 0.575 kg           |
   427→| INTERMEDIATE  | 0.325 kg           |
   428→| ADVANCED      | 0.10 kg            |
   429→
   430→#### 4.2.7 Strategy decision tree
   431→
   432→**Sex-specific boundaries** (single source of truth):
   433→
   434→| Boundary           | Male | Female |
   435→| ------------------ | ---- | ------ |
   436→| cut_floor          | 10   | 18     |
   437→| bulk_ceiling       | 20   | 28     |
   438→| bulk_start         | 15   | 23     |
   439→| operational_lo     | 10   | 18     |
   440→| operational_hi     | 20   | 28     |
   441→| obese_threshold    | 25   | 32     |
   442→| recomp_excellent   | 23   | 30     |
   443→| recomp_good_lo     | 15   | 25     |
   444→| recomp_limited     | 15   | 25     |
   445→| skinny_fat_lo      | 12   | 20     |
   446→| skinny_fat_hi      | 23   | 31     |
   447→
   448→**Algorithm (first match wins):**
   449→
   450→1. **Safety override (obesity):** `bfPct >= obese_threshold`:
   451→   - BEGINNER + goal != fat_loss → `habitChangeFirst` ("start with habit changes before calorie counting")
   452→   - Else → `cut` ("prioritise fat loss")
   453→2. **Goal = maintenance:**
   454→   - If `bfPct < hormonal_floor` → warn about bulk.
   455→   - If `bfPct > operational_hi` → warn about cut.
   456→   - **Reverse-diet detection:** if `intakeLogKcal.length >= 30` AND `operational_lo <= bfPct <= operational_hi`:
   457→     - Trailing 7-day avg intake vs estimated TDEE (compute via RMR+TDEE pipeline; fallback `weightKg × 35`).
   458→     - If `recentIntake < tdee × 0.90` → `reverseDiet`.
   459→   - Else → `maintenance`.
   460→3. **Goal = strength:** if `bfPct > bulk_ceiling` → `cut`; else `maintenance`.
   461→4. **Goal = fat_loss:** if `bfPct < cut_floor` → `maintenance` (protect hormones); else `cut`.
   462→5. **Goal = muscle_gain:** if `bfPct > bulk_ceiling` → `cut`; else `bulk`.
   463→6. **Goal = recomp:** `bfPct >= recomp_excellent` → `recomp` (excellent); `>= recomp_good_lo` → `recomp` (good); else `bulk` (limited).
   464→7. **Auto-decide (no explicit goal override — fallback path):**
   465→   - `bfPct > operational_hi` → `cut`
   466→   - `bfPct < cut_floor` → `bulk`
   467→   - Novice + skinny-fat range → `recomp`
   468→   - Novice + `bfPct < bulk_start` → `recomp` (milk newbie gains)
   469→   - `bfPct < bulk_start` → `bulk`
   470→   - Else → `cut`

   472→### 4.3 Nutrition sub-engine
   473→
   474→`buildNutritionPlan(profile, assessment, exerciseHours, exerciseIntensity, climate, weightReducedPct)`:
   475→
   476→1. **RMR** — `KATCH_MCARDLE` if `bodyFatPct != null`, else `MIFFLIN_ST_JEOR`.
   477→   - Mifflin: `10 × weight + 6.25 × height − 5 × age + (male ? 5 : -161)`
   478→   - Katch-McArdle: `370 + 21.6 × lbm` (throws if `bfPct ∉ [2, 60]`)
   479→   - Adjustments: `× 0.95` if active deficit (cut/recomp), `× 0.97` if `weightReducedPct > 0.10`.
   480→2. **TDEE** — RippedBody activity factors (1.25 / 1.45 / 1.65 / 1.85 / 2.05 for sedentary / mostly_sedentary / lightly_active / active / highly_active).
   481→3. **Adaptive TDEE** — if `weightLogKg` and `intakeLogKcal` are both non-empty, equal-length, and ≥8 entries:
   482→   - `observed = avgIntake − (Δweight × 7700) / nDays`
   483→   - Sanity: `observed ∈ [800, 7000]`, else skip.
   484→   - Ramp weight: `w = clamp((nDays − 7) / 53, 0, 1)` — no adaptation below 7 days, full at 60+.
   485→   - `adaptive = w × observed + (1 − w) × tdee`.
   486→4. **Calorie targets** — by strategy, with floor 1200 (F) / 1500 (M):
   487→
   488→| Strategy           | Computation                                                                                       |
   489→| ------------------ | ------------------------------------------------------------------------------------------------- |
   490→| cut                | `weeklyLoss = weight × rate`; `dailyDeficit = weeklyLoss × 1100`; `target = tdee − dailyDeficit` |
   491→| bulk               | `monthlyGain = weight × rate`; `dailySurplus = monthlyGain × 330`; `target = tdee + dailySurplus` |
   492→| maintenance        | `target = tdee`                                                                                   |
   493→| recomp             | `target = tdee × (1 − deficitPct)` where deficitPct = 0.15/0.05/0.0 by recomp potential           |
   494→| reverse_diet       | Weekly ladder: `+50/+100/+150 kcal/wk` by aggressiveness, until target reached                    |
   495→| habit_change_first | Same as maintenance                                                                               |
   496→
   497→**Cut rate** (priority): explicit `cutRateTier` → BF%-threshold table → default 0.75% BW/week. Hard cap 1.0% BW/week.
   498→
   499→**Cut rate by BF% threshold** (first match):
   500→
   501→| Sex    | Thresholds (BF%) | Rate (% BW/week)   |
   502→| ------ | ---------------- | ------------------ |
   503→| Male   | [25, 20, 15]     | [1.00, 0.75, 0.50, 0.50] |
   504→| Female | [32, 28, 22]     | [1.00, 0.75, 0.50, 0.50] |
   505→
   506→**Bulk rate** (priority): explicit `bulkAggressiveness × trainingStatus` table → default by status (beginner 2%/mo, novice 1.5%/mo, intermediate 1%/mo, advanced 0.5%/mo).
   507→
   508→**Bulk weekly rate tiers** (% BW/week):
   509→
   510→| Aggressiveness     | Beginner/Novice | Intermediate | Advanced |
   511→| ------------------ | --------------- | ------------ | -------- |
   512→| conservative       | 0.0020          | 0.0015       | 0.0010   |
   513→| happyMedium        | 0.0050          | 0.00325      | 0.0015   |
   514→| aggressive         | 0.0080          | 0.00575      | 0.0035   |
   515→| veryAggressive     | 0.0100          | 0.00800      | 0.0060   |
   516→
   517→5. **Macros:**
   518→   - **Protein:**
   519→     - Obese override (`bfPct >= obese_threshold`): `1 g × heightCm`.
   520→     - BF% known: `lbm_lb × 1.14` (cut) or `× 1.0` (else).
   521→     - BF% unknown: `target_weight_lb × 1.0` (cut, target = 90% current) or `weight_lb × 0.73` (else).
   522→     - Diet boost: vegan ×1.20 (ceil), vegetarian ×1.10 (ceil).
   523→   - **Fat:** midpoint of strategy range (deficit/recomp: 15-25%; surplus/maintenance/reverse: 20-30%). Floor: `max(40, weight_lb × 0.25)`. Saturated ceiling: 10% of calories.
   524→   - **Carbs:** remainder = `targetKcal − proteinKcal − fatKcal`, clamped to ≥0.
   525→6. **Hydration** (multi-step FatCalc):
   526→   - Base: `weightKg × 0.030 L`
   527→   - Sex: +0.3 L (male)
   528→   - Exercise: `hours × sweatRate / 1000` (light 300, moderate 500, intense 800 mL/hr)
   529→   - Climate multiplier (cold 0.95, temperate 1.0, hot 1.3, hot_humid 1.4) applied to exercise component only
   530→   - Pregnancy +0.3 L, breastfeeding +0.7 L (throws if male)
   531→   - Soft ceiling 5.0 L (hyponatremia protection)
   532→7. **Micronutrients:** fiber = `14 g × targetKcal / 1000`. Fruit/veg cups: 2/2 (≤2000 kcal), 3/3 (≤3000), 4/4 (≤4000+).
   533→8. **Timeline:**
   534→   - Cut: `max(4, floor(kgToLose / weeklyRateKg) + 4)` where target BF = `operational_lo + 2`.
   535→   - Bulk: `max(12, floor((targetGainKg / monthlyRateKg) × 4.348) + 4)` where targetGain = 5% BW.
   536→   - Recomp: 12 weeks. Maintenance: 12. Habit-change: 8. Reverse-diet: parse from notes, default 8.
   537→

   655→1. **Compute slot targets** from daily macros × meal allocation:
   656→

   670→- omnivore → `"OMNI"`

   675→
   676→| Component        | Weight | Tight band | Loose band |
   677→| ---------------- | ------ | ---------- | ---------- |
   678→| kcal_match       | 26     | ±20%       | ±40%       |

   680→| carb_match       | 13     | ±20%       | ±40%       |

   684→| fiber_match      | 4      | ±50%       | ±100%      |

   686→| cuisine_match    | 5      | —          | —          |

   689→- Allergen violation
   690→- Excluded ingredient present
   691→- Diet mismatch
   692→
   693→**Allergen scanning** (best-in-class — plant-qualifier suppression):
   694→- Normalize allergen aliases (`tree_nuts` → `nuts`, `crustacean(s)` → `shellfish`).

   697→- For each `dairy`/`eggs` match, check 25-char context before for plant qualifiers (almond, soy, oat, coconut, etc. — full list in §9.2). If found, skip the match.

   701→- `scaleFactor = clamp(targetKcal / recipeKcal, 0.7, 1.5)` (1.0 if within ±10%).

   705→

   749→The app owns these locally. All except `app_state` and `sync_queue` sync to Supabase.
   750→
   751→```dart
   752→@Collection()
   753→class ProfileRecord {
   754→  Id id = Isar.autoIncrement;           // always 1 (singleton per device)
   755→  late String userId;                    // Supabase auth.uid() or "anonymous"
   756→  late String dataJson;                  // serialized UserProfile + PlanPreferences
   757→  @Index() late DateTime updatedAt;
   758→  @Index() late String syncStatus;       // "pending" | "synced" | "conflict"
   759→}
   760→
   761→@Collection()
   762→class PlanRecord {
   763→  Id id = Isar.autoIncrement;
   764→  late String userId;
   765→  late String planId;                    // UUID, server-generated on first sync
   766→  late String dataJson;                  // serialized FitnessPlan
   767→  late String profileSnapshotJson;       // UserProfile at generation time
   768→  late String preferencesSnapshotJson;
   769→  late String engineVersion;             // "3.2.0"
   770→  @Index() late bool isActive;           // only one true per user
   771→  @Index() late DateTime generatedAt;
   772→  @Index() late String syncStatus;
   773→}
   774→
   775→@Collection()
   776→class WorkoutLogRecord {
   777→  Id id = Isar.autoIncrement;
   778→  late String userId;
   779→  late String planId;                    // which plan this workout belongs to
   780→  late int dayNumber;                    // 1-7 (week day)
   781→  late DateTime startedAt;
   782→  DateTime? completedAt;
   783→  late String workoutName;
   784→  late String dataJson;                  // List<SetEntry> — see below
   785→  @Index() late String syncStatus;
   786→}
   787→
   788→class SetEntry {
   789→  late String exerciseSlug;
   790→  late int setNum;
   791→  double? weightKg;
   792→  int? reps;
   793→  double? rpe;
   794→  late bool done;
   795→  late DateTime completedAt;
   796→}
   797→
   798→@Collection()
   799→class WeightLogRecord {
   800→  Id id = Isar.autoIncrement;
   801→  late String userId;
   802→  @Index() late DateTime date;
   803→  late double weightKg;                  // 30-300
   804→  @Index() late String syncStatus;
   805→}
   806→
   807→@Collection()
   808→class IntakeLogRecord {
   809→  Id id = Isar.autoIncrement;
   810→  late String userId;
   811→  @Index() late DateTime date;
   812→  late double intakeKcal;                // 0-10000
   813→  @Index() late String syncStatus;
   814→}
   815→
   816→@Collection()
   817→class AppStateRecord {
   818→  Id id = 1;                             // singleton
   819→  late String activeTab;                 // "home" | "workouts" | "meals" | "progress" | "profile"
   820→  late bool hasOnboarded;
   821→  late bool planStale;                   // profile changed since last plan generation
   822→  late String? activePlanId;
   823→}
   824→
   825→@Collection()
   826→class SyncQueueRecord {
   827→  Id id = Isar.autoIncrement;
   828→  late String operationType;             // "upsert_profile" | "upsert_plan" | "upsert_workout_log" | ...
   829→  late String recordId;                  // Isar id of the record to sync
   830→  late String collectionName;            // "ProfileRecord" | "PlanRecord" | ...
   831→  late int attempts;
   832→  late DateTime? nextAttemptAt;
   833→  @Index() late String status;           // "pending" | "in_progress" | "failed"
   834→}
   835→```
   836→
   837→**Design notes:**
   838→- `dataJson` fields store the full serialized object. Isar doesn't do nested objects well; we use it as a typed KV store for the big blobs and index only what we query by.
   839→- `syncStatus` on every record enables the background flush to find pending rows quickly.
   840→- `WorkoutLogRecord` is the daily-use loop — every set logged in a workout session persists here.
   841→- `WeightLogRecord` and `IntakeLogRecord` feed back into the engine as `weightLogKg` and `intakeLogKcal` for adaptive TDEE on the next plan generation.
   842→
   843→### 5.2 Remote (Supabase Postgres)
   844→
   845→RLS-protected. Every table has `user_id UUID REFERENCES auth.users(id)`. Every policy is `USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id)`.
   846→
   847→```sql
   848→-- profiles (1:1 with auth.users)
   849→CREATE TABLE profiles (
   850→  user_id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
   851→  data JSONB NOT NULL,
   852→  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
   853→);
   854→
   855→-- plans (1:N with users; one isActive per user)
   856→CREATE TABLE plans (
   857→  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   858→  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
   859→  data JSONB NOT NULL,                  -- full FitnessPlan
   860→  profile_snapshot JSONB NOT NULL,
   861→  preferences_snapshot JSONB NOT NULL,
   862→  engine_version TEXT NOT NULL,
   863→  is_active BOOLEAN NOT NULL DEFAULT false,
   864→  generated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
   865→  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
   866→);
   867→CREATE INDEX plans_user_active_generated_idx ON plans(user_id, is_active, generated_at DESC);
   868→CREATE INDEX plans_user_generated_idx ON plans(user_id, generated_at DESC);
   869→
   870→-- workout_logs (1:N with users + plans)
   871→CREATE TABLE workout_logs (
   872→  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   873→  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
   874→  plan_id UUID REFERENCES plans(id) ON DELETE SET NULL,
   875→  day_number INT NOT NULL,
   876→  started_at TIMESTAMPTZ NOT NULL,
   877→  completed_at TIMESTAMPTZ,
   878→  workout_name TEXT NOT NULL,
   879→  data JSONB NOT NULL,                  -- List<SetEntry>
   880→  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
   881→);
   882→CREATE INDEX workout_logs_user_started_idx ON workout_logs(user_id, started_at DESC);
   883→
   884→-- weight_logs (1:N with users)
   885→CREATE TABLE weight_logs (
   886→  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   887→  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
   888→  date DATE NOT NULL,
   889→  weight_kg NUMERIC(5,1) NOT NULL CHECK (weight_kg BETWEEN 30 AND 300),
   890→  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
   891→  UNIQUE(user_id, date)
   892→);
   893→CREATE INDEX weight_logs_user_date_idx ON weight_logs(user_id, date DESC);
   894→
   895→-- intake_logs (1:N with users)
   896→CREATE TABLE intake_logs (
   897→  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
   898→  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
   899→  date DATE NOT NULL,
   900→  intake_kcal INT NOT NULL CHECK (intake_kcal BETWEEN 0 AND 10000),
   901→  updated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
   902→  UNIQUE(user_id, date)
   903→);
   904→CREATE INDEX intake_logs_user_date_idx ON intake_logs(user_id, date DESC);
   905→```
   906→
   907→**RLS policies** (apply to every table):
   908→```sql
   909→ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;
   910→CREATE POLICY profiles_owner ON profiles
   911→  FOR ALL USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id);
   912→-- (repeat for plans, workout_logs, weight_logs, intake_logs)
   913→```
   914→
   915→**Trigger** (keep `updated_at` fresh):
   916→```sql
   917→CREATE OR REPLACE FUNCTION touch_updated_at()
   918→RETURNS TRIGGER AS $$ BEGIN NEW.updated_at = now(); RETURN NEW; END; $$ LANGUAGE plpgsql;
   919→
   920→CREATE TRIGGER profiles_touch BEFORE UPDATE ON profiles
   921→  FOR EACH ROW EXECUTE FUNCTION touch_updated_at();
   922→-- (repeat for plans, workout_logs, weight_logs, intake_logs)
   923→```
   924→
   925→### 5.3 Sync resolution
   926→
   927→- **Push:** for each `SyncQueueRecord` with `status == "pending"`:
   928→  - Read the local record.
   929→  - `UPSERT` to Supabase by `(user_id, <natural key>)` — natural key is `id` for plans/workout_logs, `date` for weight/intake logs, `user_id` for profile.
   930→  - On 200: mark local `syncStatus = "synced"`, delete queue entry.
   931→  - On 409 (conflict): fetch server row, compare `updated_at`, last-writer-wins, mark `syncStatus = "synced"` or `"conflict"` (rare — surface in UI as "Sync conflict, tap to resolve").
   932→  - On 5xx / network error: `attempts++`, `nextAttemptAt = now + min(2^attempts minutes, 1 hour)`.
   933→- **Pull:** on sign-in (and on app foreground if last pull > 1h ago):
   934→  - `SELECT * FROM <table> WHERE user_id = auth.uid() AND updated_at > <last_pull_at>`.
   935→  - For each row: if local exists, compare `updated_at`, replace if server newer. Else keep local (will be pushed next flush).
   936→  - Update `last_pull_at`.

  1101→### 6.4 Engine integration
  1102→
  1103→```dart
  1104→final engineProvider = FutureProvider<FitnEngine>((ref) async {
  1105→  final data = await loadEngineData();      // loads JSON assets
  1106→  return FitnEngine(data: data);
  1107→});
  1108→
  1109→Future<FitnessPlan> generatePlan(UserProfile profile, PlanPreferences prefs) async {
  1110→  return Isolate.run(() {
  1111→    final engine = FitnEngine(data: _engineDataSingleton);
  1112→    final assessment = engine.assessProfile(profile);
  1113→    if (assessment.isPartial) {
  1114→      throw PartialAssessmentError(assessment.errors);
  1115→    }
  1116→    return engine.proposePlan(profile, assessment, prefs);
  1117→  });
  1118→}
  1119→```
  1120→
  1121→**Why an isolate:** the engine loads 1,217 exercises + ~400 recipes + does scoring across hundreds of candidates per meal slot. On a mid-range phone, plan generation is 200-500 ms — noticeable jank if on the UI thread. Isolate keeps the UI smooth.
  1122→
  1123→**Engine data singleton:** loaded once at app startup, passed to every isolate. Don't reload per generation.

  1428→  void enqueue({required String operationType, required String recordId, required String collectionName}) {
  1429→    ref.read(syncQueueRepoProvider).enqueue(...);
  1430→    state = state.copyWith(queueLength: state.queueLength + 1);
  1431→    _flushIfNeeded();

  1435→    if (state.isFlushing) return;

  1441→        await _pushEntry(entry);
  1442→      }
  1443→      await pullAll();
  1444→    } finally {
  1445→      state = state.copyWith(isFlushing: false, lastSyncAt: DateTime.now());
  1446→    }
  1447→  }

  1472→### 8.4 Conflict resolution
  1473→
  1474→- Every record has `updated_at` (local Isar + remote Postgres, both set by trigger/app).
  1475→- On push: `INSERT ... ON CONFLICT (user_id, <natural_key>) DO UPDATE SET data = EXCLUDED.data, updated_at = now() WHERE plans.updated_at < EXCLUDED.updated_at`.
  1476→- On pull: for each server row, if local `updated_at < server.updated_at`, replace local. Else keep local (will be pushed next flush).
  1477→- **No merge** — last-writer-wins is fine for single-user, single-active-device scenarios. If the user edits on two devices simultaneously (rare), the later write wins. Surface a "sync conflict" toast if `server.updated_at > local.updated_at AND local.syncStatus == "pending"` (i.e., local has unsaved changes that would be overwritten).
  1478→
  1479→### 8.5 Offline behavior
  1480→
  1481→- All reads from Isar (local). UI never blocks on network.
  1482→- All writes to Isar + sync queue. UI updates immediately.
  1483→- Sync runs in background; failures retry with exponential backoff.
  1484→- If offline for >7 days, surface a banner: "You've been offline for a week. Sync will resume when you're back online."

  1730→### 9.5 Rounding rules
  1731→
  1732→The engine uses **banker's rounding** (round-half-to-even) everywhere.
  1733→
  1734→- `round1(0.5) == 0`, `round1(1.5) == 2`, `round1(2.5) == 2`, `round1(3.5) == 4`
  1735→- `round2(0.125) == 0.12`, `round2(0.135) == 0.14`
  1736→
  1737→**Exception:** rep-range math in DUP periodization uses `roundHalfUp` (away-from-zero on tie) — so "3-6" × 1.5 → "5-10" not "4-11". This is the only place round-half-up is used.
  1738→
  1739→Dart's `num.round()` uses round-half-up by default. **Implement banker's rounding manually:**
  1740→
  1741→```dart
  1742→double roundBankers(double value, int decimals) {
  1743→  final factor = math.pow(10, decimals);
  1744→  final scaled = value * factor;
  1745→  final floor = scaled.floor();
  1746→  final frac = scaled - floor;
  1747→  if (frac < 0.5) return floor / factor;
  1748→  if (frac > 0.5) return (floor + 1) / factor;
  1749→  // exact 0.5 → round to even
  1750→  return (floor % 2 == 0 ? floor : floor + 1) / factor;
  1751→}
  1752→
  1753→int roundBankersToInt(double value) {
  1754→  final floor = value.floor();
  1755→  final frac = value - floor;
  1756→  if (frac < 0.5) return floor;
  1757→  if (frac > 0.5) return floor + 1;
  1758→  return floor % 2 == 0 ? floor : floor + 1;
  1759→}
  1760→```
