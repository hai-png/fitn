     1тЖТ     1тЖТ# Fitn тАФ Flutter App Specification
     2тЖТ     2тЖТ
     3тЖТ     3тЖТ**Product:** Fitn тАФ a mobile fitness companion.
     4тЖТ     4тЖТ**Platform:** Flutter (iOS + Android), single Dart codebase.
     5тЖТ     5тЖТ**Audience:** Senior Flutter engineer. Build-ready.
     6тЖТ     6тЖТ
     7тЖТ     7тЖТThis document specifies a fitness app built around a deterministic, physiology-grounded engine. The engine is the product's core value тАФ its formulas, thresholds, and decision trees are pinned down completely. Everything around it (UI, state, storage, sync) is specified to a buildable level of detail, with implementation choices made where they matter and left to the engineer where they don't.
     8тЖТ     8тЖТ
     9тЖТ     9тЖТ---
    10тЖТ    10тЖТ
    11тЖТ    11тЖТ## 1. Product Vision
    12тЖТ    12тЖТ
    13тЖТ    13тЖТA mobile fitness companion that turns a 5-minute onboarding questionnaire into a fully personalized, deterministic fitness plan тАФ body composition assessment, nutrition targets, multi-week training program, 7-day meal rotation тАФ and then **stays with the user daily**: logging workouts, tracking progress, and tuning the plan as their body changes.
    14тЖТ    14тЖТ
    15тЖТ    15тЖТ**The differentiator is the engine.** It's deterministic (same inputs тЖТ byte-identical output), grounded in published physiology (Mifflin-St Jeor, Katch-McArdle, Kouri FFMI, Hodgdon-Beckett Navy, CUN-BAE, ABSI, RippedBody volume landmarks, Lyle McDonald muscle-gain rates), and produces a coherent cut/bulk/recomp/maintenance/reverse-diet recommendation with a defensible rationale. The app's job is to get out of the engine's way.
    16тЖТ    16тЖТ
    17тЖТ    17тЖТ**What the app must do well:**
    18тЖТ    18тЖТ1. **Onboard in 5 steps** тАФ minimal friction, sensible defaults, never block on optional fields.
    19тЖТ    19тЖТ2. **Generate a plan in <500 ms** тАФ instant feedback, no spinner anxiety.
    20тЖТ    20тЖТ3. **Work offline** тАФ the plan, exercise library, and recipe database live on-device. Sync is a background concern, never a UX blocker.
    21тЖТ    21тЖТ4. **Log workouts** тАФ sets, reps, weight, RPE. Rest timer. This is the daily-use loop.
    22тЖТ    22тЖТ5. **Track progress** тАФ weight log, workout history, volume trends. The engine accepts weight/intake logs as inputs for adaptive TDEE; the UI must collect them.
    23тЖТ    23тЖТ6. **Sync across devices** тАФ sign in, pick up where you left up on another phone.
    24тЖТ    24тЖТ7. **Regenerate gracefully** тАФ when the user's body changes, the plan should update without losing history.
    25тЖТ    25тЖТ
    26тЖТ    26тЖТ**What the app must NOT do:**
    27тЖТ    27тЖТ- Block on auth. Anonymous use is first-class.
    28тЖТ    28тЖТ- Require a server round-trip for plan generation.
    29тЖТ    29тЖТ- Lose data on sign-out (local data survives; only server-synced data is disowned).
    30тЖТ    30тЖТ- Pretend to be a marketplace. Commerce is out of scope.
    31тЖТ    31тЖТ

    92тЖТ    92тЖТ### 2.4 Why Supabase
    93тЖТ    93тЖТ
    94тЖТ    94тЖТSupabase provides Postgres, Auth (magic-link + OAuth), Realtime, and Storage as one service with one Flutter SDK. For a mobile app this avoids deploying, monitoring, and scaling a custom backend. The data model is plain Postgres with row-level security (RLS), so the data is portable тАФ Supabase is a hosting layer, not a proprietary lock-in.
    95тЖТ    95тЖТ
    96тЖТ    96тЖТSupabase provides:
    97тЖТ    97тЖТ- **Postgres** with row-level security тАФ no server code needed for CRUD.
    98тЖТ    98тЖТ- **Auth** with magic-link, Google, Apple, GitHub out of the box.
    99тЖТ    99тЖТ- **Realtime** (optional тАФ cross-device live sync; not needed for launch).
   100тЖТ   100тЖТ- **Storage** for any future self-hosted media (not needed тАФ YouTube embeds suffice).
   101тЖТ   101тЖТ- **Edge Functions** if server-side logic is ever needed (not needed for launch).

   135тЖТ   135тЖТ### 3.2 Engine orchestration
   136тЖТ   136тЖТ
   137тЖТ   137тЖТ```mermaid
   138тЖТ   138тЖТflowchart LR
   139тЖТ   139тЖТ  Input["UserProfile + PlanPreferences"] --> Assess["assessProfile()"]
   140тЖТ   140тЖТ  Assess --> AR["AssessmentResult<br/>(is_partial flag)"]
   141тЖТ   141тЖТ  AR -->|"is_partial == false"| Plan["proposePlan()"]
   142тЖТ   142тЖТ  Input --> Plan
   143тЖТ   143тЖТ  Plan --> NP["buildNutritionPlan()"]
   144тЖТ   144тЖТ  Plan --> TP["buildTrainingPlan()"]
   145тЖТ   145тЖТ  Plan --> MP["buildMealPlan()"]
   146тЖТ   146тЖТ  NP --> FP["FitnessPlan"]
   147тЖТ   147тЖТ  TP --> FP
   148тЖТ   148тЖТ  MP --> FP
   149тЖТ   149тЖТ  AR -->|"is_partial == true"| Err["PartialAssessmentError<br/>(UI surfaces regenerate CTA)"]
   150тЖТ   150тЖТ```
   151тЖТ   151тЖТ
   152тЖТ   152тЖТ**Critical contract:** `proposePlan()` refuses to run on a partial assessment. The UI must check `assessment.isPartial` first and show a "regenerate" CTA instead of attempting plan generation.
   153тЖТ   153тЖТ
   154тЖТ   154тЖТ### 3.3 Plan generation lifecycle
   155тЖТ   155тЖТ
   156тЖТ   156тЖТ```mermaid
   157тЖТ   157тЖТsequenceDiagram
   158тЖТ   158тЖТ  participant U as User
   159тЖТ   159тЖТ  participant UI as Flutter UI
   160тЖТ   160тЖТ  participant S as Riverpod State
   161тЖТ   161тЖТ  participant E as fitn_engine (Isolate)
   162тЖТ   162тЖТ  participant IDB as Isar
   163тЖТ   163тЖТ  participant Q as Sync Queue
   164тЖТ   164тЖТ  participant SB as Supabase
   165тЖТ   165тЖТ
   166тЖТ   166тЖТ  U->>UI: Completes onboarding step 5
   167тЖТ   167тЖТ  UI->>S: generatePlan(profile, prefs)
   168тЖТ   168тЖТ  S->>S: setGenerating(true)
   169тЖТ   169тЖТ  S->>E: Isolate.run(() => proposePlan(...))
   170тЖТ   170тЖТ  E-->>S: FitnessPlan (50-200 ms)
   171тЖТ   171тЖТ  S->>IDB: PUT plan (active=true), PUT profile
   172тЖТ   172тЖТ  S->>S: setPlan(plan), setOnboarded(true)
   173тЖТ   173тЖТ  S-->>UI: notify тЖТ render Home tab
   174тЖТ   174тЖТ  S->>Q: enqueue({type: plan, data})
   175тЖТ   175тЖТ  Q->>SB: UPSERT plan (when online)
   176тЖТ   176тЖТ  SB-->>Q: 200 OK
   177тЖТ   177тЖТ  Q->>IDB: mark synced
   178тЖТ   178тЖТ```
   179тЖТ   179тЖТ
   180тЖТ   180тЖТGeneration is local and synchronous from the UI's perspective (the isolate is an implementation detail). Sync is queued and flushed in the background. Failures are retried with exponential backoff, never shown to the user unless they persist for >24h.
   181тЖТ   181тЖТ
   182тЖТ   182тЖТ### 3.4 Sync protocol
   183тЖТ   183тЖТ
   184тЖТ   184тЖТ- **Optimistic local writes** тАФ every mutation writes to Isar immediately with `syncStatus: pending`.
   185тЖТ   185тЖТ- **Background flush** тАФ `workmanager` runs every 15 min (when online), drains the queue.
   186тЖТ   186тЖТ- **Conflict resolution** тАФ `updated_at` timestamp. Server wins if `server.updated_at > local.updated_at`. Last-writer-wins is fine for this domain (single user, single device active at a time in practice).
   187тЖТ   187тЖТ- **Push** тАФ `UPSERT` to Supabase with `WHERE user_id = auth.uid()` (RLS enforces ownership).
   188тЖТ   188тЖТ- **Pull** тАФ on sign-in, fetch all rows for the user, merge by `updated_at`.
   189тЖТ   189тЖТ- **No realtime at launch** тАФ cross-device live sync is nice-to-have, not required. Pull-on-foreground + push-on-write is sufficient.

   430тЖТ   430тЖТ#### 4.2.7 Strategy decision tree
   431тЖТ   431тЖТ
   432тЖТ   432тЖТ**Sex-specific boundaries** (single source of truth):
   433тЖТ   433тЖТ
   434тЖТ   434тЖТ| Boundary           | Male | Female |
   435тЖТ   435тЖТ| ------------------ | ---- | ------ |
   436тЖТ   436тЖТ| cut_floor          | 10   | 18     |
   437тЖТ   437тЖТ| bulk_ceiling       | 20   | 28     |
   438тЖТ   438тЖТ| bulk_start         | 15   | 23     |
   439тЖТ   439тЖТ| operational_lo     | 10   | 18     |
   440тЖТ   440тЖТ| operational_hi     | 20   | 28     |
   441тЖТ   441тЖТ| obese_threshold    | 25   | 32     |
   442тЖТ   442тЖТ| recomp_excellent   | 23   | 30     |
   443тЖТ   443тЖТ| recomp_good_lo     | 15   | 25     |
   444тЖТ   444тЖТ| recomp_limited     | 15   | 25     |
   445тЖТ   445тЖТ| skinny_fat_lo      | 12   | 20     |
   446тЖТ   446тЖТ| skinny_fat_hi      | 23   | 31     |
   447тЖТ   447тЖТ
   448тЖТ   448тЖТ**Algorithm (first match wins):**
   449тЖТ   449тЖТ
   450тЖТ   450тЖТ1. **Safety override (obesity):** `bfPct >= obese_threshold`:
   451тЖТ   451тЖТ   - BEGINNER + goal != fat_loss тЖТ `habitChangeFirst` ("start with habit changes before calorie counting")
   452тЖТ   452тЖТ   - Else тЖТ `cut` ("prioritise fat loss")
   453тЖТ   453тЖТ2. **Goal = maintenance:**
   454тЖТ   454тЖТ   - If `bfPct < hormonal_floor` тЖТ warn about bulk.
   455тЖТ   455тЖТ   - If `bfPct > operational_hi` тЖТ warn about cut.
   456тЖТ   456тЖТ   - **Reverse-diet detection:** if `intakeLogKcal.length >= 30` AND `operational_lo <= bfPct <= operational_hi`:
   457тЖТ   457тЖТ     - Trailing 7-day avg intake vs estimated TDEE (compute via RMR+TDEE pipeline; fallback `weightKg ├Ч 35`).
   458тЖТ   458тЖТ     - If `recentIntake < tdee ├Ч 0.90` тЖТ `reverseDiet`.
   459тЖТ   459тЖТ   - Else тЖТ `maintenance`.
   460тЖТ   460тЖТ3. **Goal = strength:** if `bfPct > bulk_ceiling` тЖТ `cut`; else `maintenance`.
   461тЖТ   461тЖТ4. **Goal = fat_loss:** if `bfPct < cut_floor` тЖТ `maintenance` (protect hormones); else `cut`.
   462тЖТ   462тЖТ5. **Goal = muscle_gain:** if `bfPct > bulk_ceiling` тЖТ `cut`; else `bulk`.
   463тЖТ   463тЖТ6. **Goal = recomp:** `bfPct >= recomp_excellent` тЖТ `recomp` (excellent); `>= recomp_good_lo` тЖТ `recomp` (good); else `bulk` (limited).
   464тЖТ   464тЖТ7. **Auto-decide (no explicit goal override тАФ fallback path):**
   465тЖТ   465тЖТ   - `bfPct > operational_hi` тЖТ `cut`
   466тЖТ   466тЖТ   - `bfPct < cut_floor` тЖТ `bulk`
   467тЖТ   467тЖТ   - Novice + skinny-fat range тЖТ `recomp`
   468тЖТ   468тЖТ   - Novice + `bfPct < bulk_start` тЖТ `recomp` (milk newbie gains)
   469тЖТ   469тЖТ   - `bfPct < bulk_start` тЖТ `bulk`
   470тЖТ   470тЖТ   - Else тЖТ `cut`
   471тЖТ   471тЖТ
   472тЖТ   472тЖТ### 4.3 Nutrition sub-engine
   473тЖТ   473тЖТ
   474тЖТ   474тЖТ`buildNutritionPlan(profile, assessment, exerciseHours, exerciseIntensity, climate, weightReducedPct)`:
   475тЖТ   475тЖТ
   476тЖТ   476тЖТ1. **RMR** тАФ `KATCH_MCARDLE` if `bodyFatPct != null`, else `MIFFLIN_ST_JEOR`.
   477тЖТ   477тЖТ   - Mifflin: `10 ├Ч weight + 6.25 ├Ч height тИТ 5 ├Ч age + (male ? 5 : -161)`

   479тЖТ   479тЖТ   - Adjustments: `├Ч 0.95` if active deficit (cut/recomp), `├Ч 0.97` if `weightReducedPct > 0.10`.
   480тЖТ   480тЖТ2. **TDEE** тАФ RippedBody activity factors (1.25 / 1.45 / 1.65 / 1.85 / 2.05 for sedentary / mostly_sedentary / lightly_active / active / highly_active).

   482тЖТ   482тЖТ   - `observed = avgIntake тИТ (╬Фweight ├Ч 7700) / nDays`
   483тЖТ   483тЖТ   - Sanity: `observed тИИ [800, 7000]`, else skip.
   484тЖТ   484тЖТ   - Ramp weight: `w = clamp((nDays тИТ 7) / 53, 0, 1)` тАФ no adaptation below 7 days, full at 60+.
